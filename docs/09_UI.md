# 09 UI

## UI 原则

- RPG 信息要清楚，但屏幕不能拥挤。
- 常用战斗信息固定显示。
- 背包、装备、商店使用独立面板。
- v0.4 先完成可用的会话层和显示层，不要求等待 v0.3 所有真实背包、装备、掉落数据完成。
- 当真实数据尚未接入时，UI 可以使用 `data/ui_demo_state.json` 作为演示状态；后续再替换为正式玩家数据。
- UI 逻辑不得直接依赖聊天记录里的隐含设定，NPC、商店、物品和演示状态都应来自稳定 JSON id。
- v0.4 不主动实现战斗、掉落、任务主线、职业技能、地图切换或保存读取；这些系统只预留入口。

## HUD

必须显示：

- 生命条
- 魔法条
- 经验条
- 当前等级
- 快捷栏
- 金币

基础规则：

- HUD 常驻底部，不遮挡主要战斗视野。
- 生命、魔法、经验和金币需要通过脚本接口更新，不能只写死在场景文本中。
- 快捷栏第一版使用 6 个固定槽位，先显示占位或演示物品，后续接入技能和消耗品。
- HUD 上的背包、装备按钮只负责打开对应面板，具体面板切换由统一 UI 管理节点处理。

## 背包

基础功能：

- 网格显示物品
- 鼠标悬停显示名称和属性
- 点击装备可穿戴
- 消耗品可使用

v0.4 MVP：

- 背包使用 5 列 x 6 行的 30 格网格。
- 每格显示空槽、图标、数量和选中状态。
- 鼠标悬停显示 tooltip，包含名称、品质、类型、描述、价格和出售价格。
- 第一版可以只读演示数据，不强制完成拖拽、排序、拆分堆叠或真实穿戴逻辑。

## 装备面板

槽位：

- 武器
- 衣服
- 头盔
- 项链
- 戒指

v0.4 MVP：

- 装备面板显示 5 个固定槽位。
- 装备面板必须预留角色像素半身立绘区域。
- 面板显示基础属性：攻击、防御、生命、魔法。
- 第一版可以显示演示装备和演示属性，真实装备属性计算由 v0.3 或后续接入。

## 商店

功能：

- 买入
- 卖出
- 显示金币
- 显示物品价格

v0.4 MVP：

- 商店数据来自 `data/shops.json`。
- 商品行显示名称、类型、品质、价格和购买按钮。
- 卖出视图可以先显示演示背包物品或占位列表，但界面必须预留卖出流程。
- 购买按钮第一版可以只发出信号或显示反馈，不强制修改真实背包和金币。

## 对话

NPC 对话框包含：

- NPC 名称
- NPC 头像或半身像区域
- 对话文本
- 下一句按钮
- 任务接受或完成按钮

v0.4 MVP：

- NPC 数据来自 `data/npcs.json`。
- 对话框支持逐句推进。
- 有商店的 NPC 需要显示打开商店入口。
- 任务按钮只预留位置，完整任务接受、完成和追踪逻辑属于 v0.5。

## 面板与模态规则

- 背包、装备、对话和商店属于独立面板。
- 同一时间只打开一个主要模态面板，除非后续明确需要叠放。
- 打开新面板时应关闭其它互斥面板。
- 所有面板都必须有关闭按钮。
- 后续可加入 Esc 关闭当前面板。
- 面板打开和关闭应由统一 UI Root 或菜单控制器协调，避免各面板互相直接依赖。

## 数据边界

- `data/npcs.json` 负责 NPC 名称、头像/半身像引用、对话文本和可选商店 id。
- `data/shops.json` 负责商店名称、商品列表、价格和出售价格。
- `data/ui_demo_state.json` 只用于 v0.4 UI 演示，不是正式存档。
- `data/items.json` 仍是正式物品设计数据源；当字段冲突时，应优先整理数据契约，不在 UI 中硬编码补丁。

## 风格方向

- 具体布局、尺寸、颜色和交互状态遵循 `docs/13_UIStyleGuide.md`。
- UI 使用原创像素面板，材质方向为暗木、石、青铜和皮革。
- 整体应有经典 RPG 重量感，避免现代扁平占位界面作为最终方向。
- 图标优先保持轮廓清楚，第一版可使用 `godot/assets/items/item_icons_sheet.png` 做原型显示。
- 具体尺寸、颜色和面板规范在 v0.4 中整理到 `docs/13_UIStyleGuide.md`。
